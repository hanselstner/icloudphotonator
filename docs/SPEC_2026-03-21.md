# iCloudPhotonator — Specification

- **Date:** 2026-03-21
- **Version:** 0.1.0
- **Status:** Planning

## Goal

Build **iCloudPhotonator**, a native macOS desktop tool that intelligently imports large local or network-based photo/video archives into Apple Photos for iCloud sync. The application should minimize manual work while remaining resilient, resumable, and safe for long-running imports.

## Tech Stack

| Layer | Technology |
|---|---|
| Import Engine | `osxphotos` |
| Backend | Python 3.13 |
| Persistence | SQLite |
| UI | SwiftUI |
| Communication | WebSocket JSON-RPC |
| Packaging | py2app + Xcode |

## Core Modules

- `scanner` — discovers files, classifies supported media, and identifies candidate Live Photo pairs
- `state` — defines the import job state machine and transition rules
- `job` — stores job metadata, progress counters, and resumable execution context
- `db` — manages SQLite persistence for manifests, history, and checkpoints
- `orchestrator` — coordinates the end-to-end import lifecycle
- `staging` — copies or prepares network-based assets for reliable local import
- `importer` — wraps `osxphotos import` invocations and parses import results
- `throttle` — adjusts batch sizes and cooldown timings dynamically
- `dedup` — prevents duplicate imports using manifest and import history checks
- `resilience` — handles retries, disconnect detection, and recovery behavior
- `server` — exposes the backend to the SwiftUI shell through WebSocket JSON-RPC
- `logging_config` — configures structured logging for operations and troubleshooting

## Import Workflow State Machine

The import lifecycle follows this state progression:

`IDLE → SCANNING → DEDUPLICATING → STAGING → IMPORTING → VERIFYING → COMPLETED`

At any active stage, the workflow may transition to `PAUSED` by user action or environmental constraints. Recoverable failures transition into `ERROR`, after which the system may retry and return to `IMPORTING` when conditions are safe again. This design supports controlled pauses, crash recovery, and network resilience.

## Dynamic Throttling Strategy

- Start with a batch size of **5** files
- Increase by **5** after clean successful batches
- Cap batch size at **50** files
- On error, **halve** the current batch size
- Apply a **30-second cooldown** between normal batches
- Apply an extended **2-minute cooldown every 100 files** to let Photos.app and iCloud catch up

## Supported Formats

- JPG
- JPEG
- PNG
- MOV
- MP4
- AAE

## Key Requirements

1. The user can choose a local folder or mounted network share as the import source.
2. The scanner must discover and classify supported photo/video formats before import begins.
3. The system must automatically batch imports without requiring manual batch management.
4. Deduplication must run before import and prevent obvious duplicate asset ingestion.
5. Live Photo pairs such as `HEIC+MOV` and `JPG+MOV` must be detected and preserved correctly.
6. Import progress and state must persist in SQLite so jobs can resume after crashes or restarts.
7. The user interface must expose pause, resume, and stop controls for active jobs.
8. The application must present progress counters and live logging for visibility during large imports.
9. Network-backed sources must be handled safely through staging and resilience logic when disconnects occur.
10. The tool must scale to very large archives while throttling imports so Apple Photos is not overwhelmed.

## Acceptance Criteria

- User can select any local or network folder as source
- Tool scans, deduplicates, and imports without manual batch management
- Import survives app restart and resumes from where it left off
- Network disconnect pauses import and reconnection resumes it
- Live Photo pairs import correctly
- Progress UI shows discovered, imported, skipped, duplicates, and errors
- Pause, resume, and stop controls work
- Log viewer shows real-time activity
- Works with archives of 700GB+ / 50,000+ files
- Apple Photos remains protected by throttled import behavior and cooldowns

## Risks

1. **Apple Photos rate limiting** — aggressive imports may cause Photos.app to slow down or hang, requiring conservative throttling.
2. **iCloud sync backpressure** — there is limited programmatic visibility into sync backlog, so imports may need periodic cooldowns.
3. **Database format changes** — future macOS updates may change Photos database internals used by supporting tools.
4. **Network instability** — SMB/NAS mounts may disconnect or stall, requiring timeouts, retries, and resumable state.
5. **Live Photo pairing ambiguity** — filename-based heuristics may occasionally misclassify related media pairs.

## Distribution

The product is distributed as a standalone macOS `.app` bundle that packages the SwiftUI shell and Python backend together. The bundle should be shareable with end users without requiring Python or developer tooling to be installed separately.
